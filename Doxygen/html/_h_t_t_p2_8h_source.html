244 /*****************************************************************************
245  Function:
246  HTTP_READ_STATUS HTTPReadPostPair(BYTE* cData, WORD wLen)
247 
248  Summary:
249  Reads a name and value pair from a URL encoded string in the TCP buffer.
250 
251  Description:
252  Reads a name and value pair from a URL encoded string in the TCP buffer.
253  This function is meant to be called from an HTTPExecutePost callback to
254  facilitate easier parsing of incoming data. This function also prevents
255  buffer overflows by forcing the programmer to indicate how many bytes are
256  expected. At least 2 extra bytes are needed in cData over the maximum
257  length of data expected to be read.
258 
259  This function will read until the next '&' character, which indicates the
260  end of a value parameter. It assumes that the front of the buffer is
261  the beginning of the name paramter to be read.
262 
263  This function properly updates curHTTP.byteCount by decrementing it
264  by the number of bytes read. It also removes the delimiting '&' from
265  the buffer.
266 
267  Once complete, two strings will exist in the cData buffer. The first is
268  the parameter name that was read, while the second is the associated
269  value.
270 
271  Precondition:
272  Front of TCP buffer is the beginning of a name parameter, and the rest of
273  the TCP buffer contains a URL-encoded string with a name parameter
274  terminated by a '=' character and a value parameter terminated by a '&'.
275 
276  Parameters:
277  cData - where to store the name and value strings once they are read
278  wLen - how many bytes can be written to cData
279 
280  Return Values:
281  HTTP_READ_OK - name and value were successfully read
282  HTTP_READ_TRUNCTATED - entire name and value could not fit in the buffer,
283  so input was truncated and data has been lost
284  HTTP_READ_INCOMPLETE - entire name and value was not yet in the buffer,
285  so call this function again later to retrieve
286 
287  Remarks:
288  This function is aliased to HTTPReadPostValue, since they effectively
289  perform the same task. The name is provided only for completeness.
290  ***************************************************************************/
291 #define HTTPReadPostPair(cData, wLen) HTTPReadPostValue(cData, wLen)

360 /*****************************************************************************
361  Function:
362  HTTP_IO_RESULT HTTPExecutePost(void)
363 
364  Summary:
365  Processes POST form variables and data.
366 
367  Description:
368  This function is implemented by the application developer in
369  CustomHTTPApp.c. Its purpose is to parse the data received from
370  POST forms and perform any application-specific tasks in response
371  to these inputs. Any required authentication has already been
372  validated before this function is called.
373 
374  When this function is called, POST data will be waiting in the
375  TCP buffer. curHTTP.byteCount will indicate the number of bytes
376  remaining to be received before the browser request is complete.
377 
378  Since data is still in the TCP buffer, the application must call
379  TCPGet or TCPGetArray in order to retrieve bytes. When this is done,
380  curHTTP.byteCount MUST be updated to reflect how many bytes now
381  remain. The functions TCPFind, TCPFindString, TCPFindROMString,
382  TCPFindArray, and TCPFindROMArray may be helpful to locate data
383  in the TCP buffer.
384 
385  In general, data submitted from web forms via POST is URL encoded.
386  The HTTPURLDecode function can be used to decode this information
387  back to a standard string if required. If data buffer space
388  associated with this connection is required, curHTTP.data may be
389  overwritten here once the application is done with the values.
390  Any data placed there will be available to future callbacks for
391  this connection, including HTTPExecutePost and any
392  HTTPPrint_varname dynamic substitutions.
393 
394  Whenever a POST form is processed it is recommended to issue a
395  redirect back to the browser, either to a status page or to
396  the same form page that was posted. This prevents accidental
397  duplicate submissions (by clicking refresh or back/forward) and
398  avoids browser warnings about "resubmitting form data". Redirects
399  may be issued to the browser by setting curHTTP.data to the
400  destination file or URL, and curHTTP.httpStatus to HTTP_REDIRECT.
401 
402  Finally, this function may set cookies. Set curHTTP.data to a series
403  of name/value string pairs (in the same format in which parameters
404  arrive) and then set curHTTP.hasArgs equal to the number of cookie
405  name/value pairs. The cookies will be transmitted to the browser,

407  curHTTP.data.

420  process to complete, and this function should
421  be called again later
422 
423  Remarks:
424  This function is only called when the request method is POST, and is
425  only used when HTTP_USE_POST is defined. This method may NOT write

529  void HTTPPrint_varname(void)
530  void HTTPPrint_varname(WORD wParam1)
531  void HTTPPrint_varname(WORD wParam1, WORD wParam2, ...)
532 
533  Summary:
534  Inserts dynamic content into a web page
535 
536  Description:
537  Functions in this style are implemented by the application developer in
538  CustomHTTPApp.c. These functions generate dynamic content to be
539  inserted into web pages and other files returned by the HTTP2 server.
540 
541  Functions of this type are called when a dynamic variable is located
542  in a web page. (ie, ~varname~ ) The name between the tilde '~'
543  characters is appended to the base function name. In this example, the
544  callback would be named HTTPPrint_varname.
545 
546  The function prototype is located in your project's HTTPPrint.h, which
547  is automatically generated by the MPFS2 Utility. The prototype will
548  have WORD parameters included for each parameter passed in the dynamic
549  variable. For example, the variable "~myArray(2,6)~" will generate the
550  prototype "void HTTPPrint_varname(WORD, WORD);".
551 
552  When called, this function should write its output directly to the TCP
553  socket using any combination of TCPIsPutReady, TCPPut, TCPPutArray,
554  TCPPutString, TCPPutROMArray, and TCPPutROMString.
555 
556  Before calling, the HTTP2 server guarantees that at least
557  HTTP_MIN_CALLBACK_FREE bytes (defaults to 16 bytes) are free in the
558  output buffer. If the function is writing less than this amount, it
559  should simply write the data to the socket and return.
560 
561  In situations where a function needs to write more this amount, it
562  must manage its output state using curHTTP.callbackPos. This value
563  will be set to zero before the function is called. If the function is
564  managing its output state, it must set this to a non-zero value before
565  returning. Typically this is used to track how many bytes have been
566  written, or how many remain to be written. If curHTTP.callbackPos is
567  non-zero, the function will be called again when more buffer space is
568  available. Once the callback completes, set this value back to zero
569  to resume normal servicing of the request.
570 
571  Precondition:
572  None
573 
574  Parameters:
575  wParam1 - first parameter passed in the dynamic variable (if any)
576  wParam2 - second parameter passed in the dynamic variable (if any)
577  ... - additional parameters as necessary
578 
579  Returns:
580  None
581 
582  Remarks:
583  This function may service multiple HTTP requests simultaneously,
584  especially when managing its output state. Exercise caution when using
585  global or static variables inside this routine. Use curHTTP.callbackPos
586  or curHTTP.data for storage associated with individual requests.
