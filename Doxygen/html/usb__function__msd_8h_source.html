474  LUN_FUNCTIONS is a structure of function pointers that tells the stack
475  where to find each of the physical layer functions it is looking for.
476  This structure needs to be defined for any project for PIC24F or PIC32.
477 
478  Typical Usage:
479  <code>
480  LUN_FUNCTIONS LUN[MAX_LUN + 1] =
481  {
482  {
483  &amp;MDD_SDSPI_MediaInitialize,

488  &amp;MDD_SDSPI_WriteProtectState,
489  &amp;MDD_SDSPI_SectorWrite
490  }
491  };
492  </code>
493 
494  In the above code we are passing the address of the SDSPI functions to
495  the corresponding member of the LUN_FUNCTIONS structure. In the above
496  case we have created an array of LUN_FUNCTIONS structures so that it is
497  possible to have multiple physical layers by merely increasing the
498  MAX_LUN variable and by adding one more set of entries in the array.
499  Please take caution to insure that each function is in the the correct
500  location in the structure. Incorrect alignment will cause the USB stack
501  to call the incorrect function for a given command.

504  available physical media, their requirements, and how to use their
505  associated functions.

550 /**************************************************************************
551  Function: BOOL MSDWasLastCBWValid(void)
552 
553  Summary:
554  Returns the BOOLean status of the most recently received command block
555  wrapper (CBW). If the last CBW passed the MSD "is valid" tests, then
556  this function will return TRUE. If the last received CBW was
557  not valid, then the return value will be FALSE.
558 
559  Description:
560  Returns the BOOLean status of the most recently received command block
561  wrapper (CBW). If the last CBW passed the "is valid" tests, then
562  this function will return TRUE. If the last received CBW was
563  not valid, then the return value will be FALSE. This function would
564  typically be used following a host initiated clear endpoint halt
565  operation on an MSD bulk IN or OUT endpoint. In this case, the firmware
566  needs to check if the last received CBW was valid, in order to know
567  if the firmware should re-stall the endpoint following the clear halt.
568  This is necessary to fully comply with the MSD BOT error case handling
569  related specifications, which dicate that the MSD device must
570  persistently STALL the bulk endpoints when a non-valid CBW is received.
571  The host must issue an MSD reset command over EP0, prior to clear
572  endpoint halt, in order to clear the stall condition in this special
573  scenario.
574 
575  Parameters:
576  None
577 
578  Return Values:
579  BOOL: TRUE - if last received CBW was valid
580  FALSE - if the last received CBW was not valid
581 
582  Remarks:
583  None
584
