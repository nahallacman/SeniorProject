173  //:::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::
174  // These configuration options define the callback functions used by mComm to allow custom
175  // commands and actions to be implemented by the application.
176  //
177  // These are only used if MCOMM_ENABLE_CUSTOM_OPCODE is defined.
178  //
179  // An example custom packet is implemented in mComm_custom.c - it creates an onChange
180  // notification packet that outputs the new state mask if any sensors have been updated. This
181  // is used by the mTouch GUI, but may be manipulated to implement any custom action for your
182  // application.
183  //
184  // There are ## steps to implementing a custom packet:
185  // 1. Enable the #define for MCOMM_ENABLE_CUSTOM_OPCODE
186  // 2. Create a callback function to notify mComm if you have data to send.
187  // 3. Create a process function to initialize the mComm output vector.
188  // An example is provided in mComm_custom.c. You can also see how the mComm implements
189  // its RAM, NVM, and Stream process functions in mComm_opcodes.c
190  // 4. Create a read and/or write iterator function.
191  // Read iterator functions should return the next byte to be output by the comms.
192  // Write iterator functions should write one byte each call and should clear the
193  // 'hasNext' bit when finished.
194  //
195  // The 'callback' function is used by mComm to see if any custom data is ready to be sent.
196  // This function should return a 1 if you wish to send data, and a 0 if not.
197  // If the 'callback' function returns a 1, mComm will eventually call the 'process' function.
198  // (Input packets have a higher priority. If one is waiting to be processed, the callback
199  // function may be called several times before 'process' is called.)
200  // The 'process' function is called once the custom packet has priority and is scheduled to be
201  // output next. During this function, all output buffer variables should be initialized.
202  // Finally, mComm will call your 'iterator' function each time it requires a new byte. The
203  // 'read' iterator function should iterate through its output packet and return the next
204  // byte to be sent. Ex: The first time it's called, it may return a packet byte length.
205  // The next time it returns an opcode, then it returns the first payload byte, etc.
206  // By contrast, the 'write' iterator function will iterate over the input buffer and
207  // write its data to the location in memory pointed to by the output vector.
208  // With your custom command, mComm will use the 'read' iterator function anytime 'callback'
209  // returns a 1 or the master uses the 'read custom' opcode. mComm will use the 'write'
210  // iterator only when the master uses the 'write custom' opcode.
211  //
212  // To stop mComm from calling the iterator function and to end the packet,
213  // mComm_output.flags.bits.hasNext should be cleared.
214  //
215  // The mComm code comes with iterators to read and write RAM and EEPROM by default. Using
216  // these for your custom packets will allow for a more efficient result. The example
217  // onChange packet in mComm_custom.c uses the ramReadIterator() function with a custom
218  // buffer.
219  //:::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::::

413  //========================================================================
414  // mComm Configuration Block Values
415  //========================================================================
416  // Two configuration blocks are implemented in constant memory to store
417  // the current configuration of the system. Which values are stored depends
418  // on the following values.
419  //
420  // This data is accessible by the master by reading from RAM locations:
421  // 0x0000 :: Literals
422  // 0x0001 :: Addresses
423  //
424  // These configuration blocks can then tell the master where the important
425  // RAM arrays are located that it might wish to read/write to.
426  //
427  // Example:
428  // * The master wishes to read the current reading for each sensor.
429  // 1. The master will request to read from RAM location 0x0000.
430  // The PIC will respond with the values in MCOMM_CONFIG_LIT_INIT.
431  // The master then takes the 4th byte and knows the number of sensors ("X").
432  // 2. The master will request to read from RAM location 0x0001.
433  // The PIC will respond with the values in MCOMM_CONFIG_ADDR_INIT.
434  // The master then takes the 5th address and knows where the raw
435  // data array is being stored in RAM ("Y").
436  // 3. The master will then request to read 2*X bytes from RAM location Y.
437  //
438  // NOTE: Steps 1 & 2 only need to be performed once.
439
