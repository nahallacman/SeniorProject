185 /* Isochronous Data Buffer
186 
187 Isochronous data transfers are continuous, until they are explicitly terminated.
188 The maximum transfer size is given in the endpoint descriptor, but a single
189 transfer may contain less data than the maximum. Also, the USB peripheral can
190 store data to RAM in a linear fashion only. Therefore, we cannot use a simple
191 circular buffer for the data. Instead, the application or client driver must
192 allocate multiple independent data buffers. These buffers must be the
193 maximum transfer size. This structure contains a pointer to an allocated
194 buffer, plus the valid data length of the buffer.
195 */

554 /****************************************************************************

556  BOOL USBHostDeviceSpecificClientDriver( BYTE deviceAddress )
557 
558  Summary:
559  This function indicates if the specified device has explicit client
560  driver support specified in the TPL.
561 
562  Description:
563  This function indicates if the specified device has explicit client
564  driver support specified in the TPL. It is used in client drivers'
565  USB_CLIENT_INIT routines to indicate that the client driver should be
566  used even though the class, subclass, and protocol values may not match
567  those normally required by the class. For example, some printing devices
568  do not fulfill all of the requirements of the printer class, so their
569  class, subclass, and protocol fields indicate a custom driver rather than
570  the printer class. But the printer class driver can still be used, with
571  minor limitations.
572 
573  Precondition:
574  None
575 
576  Parameters:
577  BYTE deviceAddress - Address of device
578 
579  Return Values:
580  TRUE - This device is listed in the TPL by VID andPID, and has explicit
581  client driver support.
582  FALSE - This device is not listed in the TPL by VID and PID.
583 
584  Remarks:
585  This function is used so client drivers can allow certain
586  devices to enumerate. For example, some printer devices indicate a custom
587  class rather than the printer class, even though the device has only minor
588  limitations from the full printer class. The printer client driver will
589  fail to initialize the device if it does not indicate printer class support
590  in its interface descriptor. The printer client driver could allow any
591  device with an interface that matches the printer class endpoint
592  configuration, but both printer and mass storage devices utilize one bulk
593  IN and one bulk OUT endpoint. So a mass storage device would be
594  erroneously initialized as a printer device. This function allows a
595  client driver to know that the client driver support was specified
596  explicitly in the TPL, so for this particular device only, the class,
597  subclass, and protocol fields can be safely ignored.
